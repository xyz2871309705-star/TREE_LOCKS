# 分布式树状锁管理器 设计文档

> 版本: 0.1.0  
> 日期: 2026-06-12  
> 状态: 草案

---

## 1. 概述

### 1.1 目标

实现一套 **分布式多粒度树状锁** 系统，为通用 N 叉树结构提供：

- **一致性保护**：确保子树操作期间，祖先节点状态不被并发修改破坏
- **死锁预防**：通过锁层级协议在协议层面消除死锁
- **分布式协调**：多个进程/节点之间安全地共享树锁

### 1.2 核心思想

采用数据库领域经典的 **多粒度意向锁（Multiple Granularity Locking with Intention Locks）** 协议，将锁的粒度和树结构自然对齐。

### 1.3 适用范围

| 场景 | 说明 |
|------|------|
| 分布式文件系统 | 目录树的并发读写控制 |
| 分布式数据库索引 | B-tree / LSM-tree 的页级并发 |
| 分布式配置管理 | 层级配置项的并发修改 |
| 协同文档编辑 | JSON/XML 文档树的并发编辑 |

## 3. 锁协议设计

### 3.1 锁模式

TreeLocks 实现了经典的 **Gray 6-mode 多粒度意向锁协议**。这 6 种模式按语义分为两组：

| 分组 | 包含模式 | 语义 |
|------|---------|------|
| **意向锁** | `IS`, `IX`, `SIX` | 放在**祖先节点**上，声明子树里发生了什么操作 |
| **实际锁** | `S`, `X` | 放在**目标节点**上，真正保护该子树的数据读写 |

`TREELOCK_NL` 是特殊的"无锁"状态；`TREELOCK_SIX` 是意向锁与实际锁的组合态。

#### 概览表

| 枚举值 | 值 | 符号 | 含义 |
|--------|----|------|------|
| `TREELOCK_NL` | 0 | NL | 无锁 |
| `TREELOCK_IS` | 1 | IS | 意向共享：子树中某节点持有 S |
| `TREELOCK_IX` | 2 | IX | 意向排他：子树中某节点持有 X |
| `TREELOCK_S` | 3 | S | 共享：读该节点及其整棵子树 |
| `TREELOCK_SIX` | 4 | SIX | S + IX：读整棵子树，同时某子节点正被排他写 |
| `TREELOCK_X` | 5 | X | 排他：独占该节点及其整棵子树 |

#### 逐项说明

**1. `TREELOCK_NL` — 无锁（No Lock）**

- 节点上没有任何锁，或不持锁的状态。
- 查询 `treelock_get_mode()` 时，未持有锁的节点返回此值。
- 兼容矩阵中，NL 与所有模式兼容（不阻塞任何人）。
- 在锁降级路径中，所有模式都可以降级到 NL（即释放锁）。

**2. `TREELOCK_IS` — 意向共享（Intention Shared）**

- 声明"我的某个子孙节点持有 S 锁"，即**子树中某节点正在被读取**。
- 持有 IS 的节点本身不能被写（IS 与 X 不兼容），但子树内可以自由加 S 锁。
- 加锁协议要求：在节点 N 上获取 IS 前，N 的父节点必须持有 IS 或更强锁。
- 典型场景：读一个目录的第 3 层文件时，根节点和中间目录都需要加 IS（宣告"子树里有读者"）。

**3. `TREELOCK_IX` — 意向排他（Intention Exclusive）**

- 声明"我的某个子孙节点持有 X 锁"，即**子树中某节点正在被排他写入**。
- 持有 IX 的节点本身不能被读（IX 与 S/SIX 不兼容），但子树内可以自由加 X 锁。
- 加锁协议要求：在节点 N 上获取 IX 前，N 的父节点必须持有 IX 或更强锁。
- 典型场景：写一个子文件时，根节点和父目录都需要加 IX（宣告"子树里有写者"）。

**4. `TREELOCK_S` — 共享（Shared）**

- **读取该节点及其整棵子树的全部内容**。多个客户端可以同时持有 S 锁（S 与 S 兼容）。
- 兼容性：与 `IS`/`NL` 兼容；与 `IX`/`SIX`/`X` **不兼容**（保证写者进不来）。
- 加锁协议要求：父节点必须持有 IS 或更强锁。
- 典型场景：批量读取目录及所有子文件（备份、统计等）。

**5. `TREELOCK_SIX` — 共享意向排他（Shared + Intention Exclusive）**

- **S + IX 的组合**：同时拥有 S 的"读整棵子树"能力和 IX 的"子树中有写者"声明。
- 这意味着：**"我读整棵树，但我也在修改其中的某个子节点"**。
- 兼容性：因同时具有 S 和 IX 的特性，只与 `NL` 和 `IS` 兼容（限制与 X 接近）。
- 加锁协议要求：父节点必须持有 IX 或更强锁。
- 典型场景：遍历目录树做批量更新——需要看全貌（S），又要改其中某个文件（IX）。

**6. `TREELOCK_X` — 排他（Exclusive）**

- **独占该节点及其整棵子树**，读写都不允许其他人触碰。
- 兼容性：只与 `NL` 兼容，拒绝一切其他模式的请求。
- 加锁协议要求：父节点必须持有 IX 或更强锁。
- 典型场景：删除整棵子树、重命名目录、全量替换子树内容。

#### 锁强度偏序

```
NL → IS → IX → S/SIX → X
```

其中：
- **S 和 IX 不可比**（互不兼容）—— S 允许并发读但排斥写意图，IX 允许写意图但排斥读。
- **SIX 介于 IX 和 X 之间**——比 IX 多了读整棵子树的能力，比 X 弱（仍允许其他客户端的 IS 锁）。

代码中体现为 `g_required_parent` 表（`protocol.c`）：

| 请求模式 | 父节点最小锁 | 原因 |
|---------|------------|------|
| NL | NL | 不需要锁 |
| IS | IS | 读操作的意向声明 |
| S | IS | 读操作，需要父节点宣告"子树有读者" |
| IX | IX | 写操作的意向声明 |
| SIX | IX | 含写操作，需要父节点宣告"子树有写者" |
| X | IX | 写操作，需要父节点宣告"子树有写者" |

### 3.2 兼容矩阵

```
         │  NL   IS   IX    S   SIX   X
    ─────┼───────────────────────────────
     NL  │  ✅   ✅   ✅   ✅   ✅   ✅
     IS  │  ✅   ✅   ✅   ✅   ✅   ❌
     IX  │  ✅   ✅   ✅   ❌   ❌   ❌
     S   │  ✅   ✅   ❌   ✅   ❌   ❌
     SIX │  ✅   ✅   ❌   ❌   ❌   ❌
     X   │  ✅   ❌   ❌   ❌   ❌   ❌
```

### 3.3 锁强度偏序

```
锁强度从弱到强：
  NL → IS → IX → S/SIX → X

其中 S 和 IX 不可比（互不兼容）
     SIX 介于 IX 和 X 之间
```

### 3.4 加锁协议

```
规则 1（自顶向下）：
  在节点 N 上获取 S 或 IS 锁之前，
  必须持有 N 的父节点的 IS 或 IX（或更强）锁。

规则 2（自顶向下）：
  在节点 N 上获取 X、IX 或 SIX 锁之前，
  必须持有 N 的父节点的 IX 或 SIX（或更强）锁。

规则 3（自底向上）：
  释放锁时，必须先释放子节点的锁，再释放父节点的锁。

规则 4（根节点）：
  根节点没有父节点，可以直接获取任何模式的锁。
```

### 3.5 锁升级 / 降级

```
允许的升级路径（持有旧锁 → 申请更强锁）：
  NL → IS → S
  NL → IS → IX → X
  NL → IS → IX → SIX → X
  IS → IX  (允许，但必须先释放 S)
  S  → SIX → X
  IX → SIX → X

允许的降级路径（释放后重新获取更弱锁）：
  X → SIX → S 或 IX
  X → IS → NL
  SIX → S 或 IX
```

## 4. C 客户端 API 设计

### 4.1 数据类型

```c
/* 锁模式 */
typedef enum {
    TREELOCK_NL  = 0,  /* No Lock          */
    TREELOCK_IS  = 1,  /* Intention Shared */
    TREELOCK_IX  = 2,  /* Intention Exclusive */
    TREELOCK_S   = 3,  /* Shared           */
    TREELOCK_SIX = 4,  /* Shared + Intention Exclusive */
    TREELOCK_X   = 5,  /* Exclusive        */
} treelock_mode_t;

/* 树节点标识（应用自定义，例如路径哈希或 UUID） */
typedef uint64_t treelock_node_id_t;

/* 锁句柄 */
typedef struct treelock_s treelock_t;

/* 客户端配置 */
typedef struct {
    const char *server_addr;    /* 服务端地址 "ip:port" */
    int         timeout_ms;     /* 默认超时 */
    const char *client_id;      /* 客户端标识 */
} treelock_config_t;
```

### 4.2 核心 API

```c
/* ========== 生命周期 ========== */

/* 创建锁客户端实例，连接服务端 */
treelock_t *treelock_create(const treelock_config_t *config);

/* 销毁客户端，释放所有持有的锁 */
void treelock_destroy(treelock_t *tl);

/* ========== 锁操作 ========== */

/* 阻塞获取锁，直到成功或出错 */
int treelock_lock(
    treelock_t         *tl,
    treelock_node_id_t  node_id,
    treelock_mode_t     mode
);

/* 尝试获取锁，超时返回 TREELOCK_ERR_TIMEOUT */
int treelock_try_lock(
    treelock_t         *tl,
    treelock_node_id_t  node_id,
    treelock_mode_t     mode,
    int                 timeout_ms
);

/* 释放指定节点的锁 */
int treelock_unlock(
    treelock_t         *tl,
    treelock_node_id_t  node_id
);

/* 释放该客户端持有的所有锁 */
int treelock_unlock_all(treelock_t *tl);

/* ========== 锁升级 / 降级 ========== */

/* 将节点上已有锁升级为新模式（阻塞） */
int treelock_escalate(
    treelock_t         *tl,
    treelock_node_id_t  node_id,
    treelock_mode_t     new_mode
);

/* 将节点上已有锁降级为新模式 */
int treelock_downgrade(
    treelock_t         *tl,
    treelock_node_id_t  node_id,
    treelock_mode_t     new_mode
);

/* ========== 查询 ========== */

/* 查询当前客户端在指定节点上持有的锁模式 */
treelock_mode_t treelock_get_mode(
    treelock_t         *tl,
    treelock_node_id_t  node_id
);

/* 查询指定节点上所有客户端持有的锁（调试用） */
int treelock_query_node(
    treelock_t         *tl,
    treelock_node_id_t  node_id,
    /* out */ char    **json_result
);

/* ========== 回调 ========== */

/* 锁被抢占时的回调（租约过期等异常） */
typedef void (*treelock_lost_cb)(
    treelock_node_id_t node_id,
    treelock_mode_t    held_mode,
    void              *user_data
);

int treelock_set_lost_callback(
    treelock_t     *tl,
    treelock_lost_cb cb,
    void           *user_data
);
```

### 4.3 返回值

```c
#define TREELOCK_OK            0
#define TREELOCK_ERR_TIMEOUT  -1   /* 超时未获取到锁 */
#define TREELOCK_ERR_CONFLICT -2   /* 兼容冲突 */
#define TREELOCK_ERR_PROTOCOL -3   /* 违反加锁协议 */
#define TREELOCK_ERR_NETWORK  -4   /* 网络/连接错误 */
#define TREELOCK_ERR_STALE    -5   /* 租约过期，锁已丢失 */
#define TREELOCK_ERR_INVAL    -6   /* 参数无效 */
```

### 4.4 使用示例

```c
/* 示例：对树进行路径加锁，读目录 + 写文件 */
void update_file(treelock_t *tl,
                 treelock_node_id_t root,
                 treelock_node_id_t dir,
                 treelock_node_id_t file)
{
    /* 自顶向下加锁 */
    treelock_lock(tl, root, TREELOCK_IS);   // 根: 意向共享
    treelock_lock(tl, dir,  TREELOCK_IS);   // 目录: 意向共享（读目录内容）
    treelock_lock(tl, file, TREELOCK_X);    // 文件: 排他（写文件）

    /* ... 执行写操作 ... */

    /* 自底向上释放 */
    treelock_unlock(tl, file);
    treelock_unlock(tl, dir);
    treelock_unlock(tl, root);
}

/* 示例：锁定整棵子树进行批量操作 */
void batch_subtree(treelock_t *tl,
                   treelock_node_id_t parent,
                   treelock_node_id_t subtree_root)
{
    /* 先持有父节点的 IX */
    treelock_lock(tl, parent, TREELOCK_IX);
    /* 锁住整棵子树 */
    treelock_lock(tl, subtree_root, TREELOCK_X);

    /* ... 批量修改子树 ... */

    treelock_unlock(tl, subtree_root);
    treelock_unlock(tl, parent);
}
```

## 5. 服务端设计

### 5.1 锁表结构

```
LockTable {
    NodeLock[node_id] {
        granted_locks: [
            { client_id, mode, lease_expire_at }
        ]
        wait_queue: [
            { client_id, requested_mode, enqueue_time }
        ]
    }
}
```

### 5.2 租约机制

```
每个锁授予时附带租约 TTL（默认 30 秒）
客户端需定期心跳续约（每 TTL/3 续约一次）
服务端租约超时 → 自动释放锁 → 通知等待队列中的下一个客户端
客户端心跳失败 → 触发 treelock_lost_cb 回调
```

### 5.3 死锁处理

本协议**不依赖死锁检测**，通过以下机制天然防死锁：

1. **全局锁顺序**：所有客户端必须自顶向下加锁 → 消除循环等待
2. **模式兼容矩阵**：不兼容的锁请求进入 FIFO 等待队列
3. **租约超时**：防止客户端崩溃导致永久阻塞

### 5.4 故障恢复

```
锁管理器崩溃恢复：
  1. Raft 日志重放 → 恢复锁表状态
  2. 所有客户端重新连接 → 重新获取锁
  3. 冲突时：先到先得，后到的返回 TREELOCK_ERR_STALE

网络分区处理：
  1. 分区侧客户端租约过期 → 自动释放锁
  2. 多数派侧继续服务
  3. 分区恢复后客户端重新建立 session
```

## 6. 通信协议

### 6.1 消息格式（protobuf 定义）

```protobuf
syntax = "proto3";

message LockRequest {
    string client_id       = 1;
    uint64 request_id      = 2;
    uint64 node_id         = 3;
    int32  mode            = 4;  // 0=NL, 1=IS, 2=IX, 3=S, 4=SIX, 5=X
    int32  timeout_ms      = 5;  // 0 = 阻塞等待
}

message LockResponse {
    uint64 request_id      = 1;
    int32  result          = 2;  // 0=OK, 负值=错误码
    uint64 lease_expire_at = 3;  // Unix 时间戳 ms
}

message UnlockRequest {
    string client_id       = 1;
    uint64 node_id         = 2;
}

message Heartbeat {
    string client_id       = 1;
    repeated uint64 node_ids = 2;  // 需要续约的节点列表
}
```

### 6.2 连接模型

```
客户端 → 服务端: 单一 TCP 长连接
  - 多路复用：一个连接承载所有节点的锁请求
  - 请求-响应 + 服务端推送（租约过期通知）
  - 连接断开 → 客户端本地缓存锁失效 → 等待重连后重新获取
```

## 9. 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 客户端缓存锁状态？ | 是，缓存已获取的锁 | 减少服务端往返 |
| 锁释放通知？ | 推送通知（ZK watch/服务端推送） | 减少轮询开销 |
| 锁粒度？ | 由 `node_id` 定义，库不关心节点内容 | 保持通用性 |
| 节点增删？ | 由应用负责，锁前确保节点存在 | 锁管理器不维护树结构 |
| 租约 vs 心跳？ | 租约（服务端倒计时） | 比心跳更可靠，防止网络抖动 |
| 超时处理？ | 不自动重试，返回错误给调用者 | 让应用决定重试策略 |

## 10. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 锁管理器成为性能瓶颈 | 高延迟 | 批量操作 API、客户端缓存 |
| 网络分区导致脑裂 | 数据不一致 | Raft 多数派 + 租约 fencing token |
| 客户端崩溃锁不释放 | 资源泄漏 | 租约超时自动回收 |
| 锁升级导致死锁 | 系统 hang | 协议保证自顶向下，不允许逆序升级 |
| ZK 依赖过重 | 部署复杂 | 阶段一先做单机版，不依赖 ZK |
