## 1. 现状分析

### 1.1 已有的能力

| 能力 | 文件 | 状态 |
|------|------|------|
| 6 种锁模式 + 兼容矩阵 | `protocol.c` | ✅ 完成 |
| FIFO 等待队列 + 条件变量 | `lock_table.c` | ✅ 完成 |
| 线程安全 lock/unlock/try_lock | `client.c` | ✅ 完成 |
| 锁升级/降级路径验证 | `client.c` | ✅ 完成 |
| 引用计数（同客户端重入） | `client.c` | ✅ 完成 |
| 协议校验函数声明 | `internal.h:214` | ✅ 已声明，**尚未实现** |

### 1.2 缺失的能力

| 能力 | 说明 |
|------|------|
| **父子关系注册** | `treelock_node_t` 中没有 `parent_id` 字段 |
| **树拓扑存储** | `treelock_s` 中没有树结构的索引 |
| **协议规则自动校验** | `treelock_validate_protocol()` 仅声明，函数体不存在 |
| **从文件加载树** | 无 JSON 解析器，无文件加载 API |
| **路径加锁** | 用户必须手动逐节点加锁 |

### 1.3 关键发现

`internal.h` 中已经声明了 `treelock_validate_protocol()` 和 `treelock_required_parent_mode()`，说明协议校验的接口已经在设计阶段预留好了。`lock_table.c` 中锁表是**单链表**（O(n) 查找），ROADMAP 提到后续要升级为 hash 表。

**结论**：当前代码架构为树结构管理留好了接口，但核心数据结构需要扩展。这是增量开发，不是重构。

## 2. 方案概览

```
┌─────────────────────────────────────────────────────────────┐
│                      用户视角                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   1. 写 JSON 文件描述树结构                                   │
│                                                             │
│   2. treelock_create()          ← 创建客户端（不变）          │
│   3. treelock_load_tree(tl, f)  ← 一行加载树（新增）          │
│   4. treelock_lock_path(tl, p)  ← 一行路径加锁（新增）         │
│   5. treelock_unlock_path(tl, p)← 一行路径解锁（新增）         │
│   6. treelock_destroy()         ← 释放资源（不变）            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**核心设计原则**：
- **增量**：不修改现有 public API 签名，所有新功能作为新增接口
- **轻量**：解析器选 cJSON（~1000 行 C，MIT 协议，单文件即可集成）
- **分层**：L1（树注册）→ L2（路径加锁）→ L3（操作声明文件），每层独立可用

## 3. 文件格式规范

### 3.1 模式 A：扁平表示（推荐用于大批量节点）

```json
{
  "version": 1,
  "name": "filesystem_tree",
  "root_id": 1,
  "nodes": [
    { "id": 1,  "label": "/" },
    { "id": 2,  "label": "home",   "parent": 1 },
    { "id": 3,  "label": "var",    "parent": 1 },
    { "id": 10, "label": "alice",  "parent": 2 },
    { "id": 11, "label": "bob",    "parent": 2 },
    { "id": 20, "label": "log",    "parent": 3 },
    { "id": 21, "label": "cache",  "parent": 3 }
  ]
}
```

**规则**：
- `id` 必须是 `uint64_t` 范围内的正整数
- `parent` 指向父节点的 `id`，根节点的 `parent` 为 `0` 或省略
- `label` 是字符串，用于路径查找（如 `/home/alice`）
- 所有 `id` 必须唯一
- 所有 `parent` 必须指向已存在的节点（或 0）
- 必须有且仅有一个根节点（`parent == 0`）
- 不允许有环

### 3.2 模式 B：嵌套表示（推荐用于手工编写）

```json
{
  "version": 1,
  "name": "filesystem_tree",
  "tree": {
    "id": 1,
    "label": "/",
    "children": [
      {
        "id": 2,
        "label": "home",
        "children": [
          { "id": 10, "label": "alice" },
          { "id": 11, "label": "bob" }
        ]
      },
      {
        "id": 3,
        "label": "var",
        "children": [
          { "id": 20, "label": "log" },
          { "id": 21, "label": "cache" }
        ]
      }
    ]
  }
}
```

### 3.3 解析策略

两种格式在内部统一转换为模式 A 的扁平结构。判断逻辑：
- JSON 中存在 `"tree"` 键 → 嵌套模式，递归展开
- JSON 中存在 `"nodes"` 键 → 扁平模式，直接解析
- 两者都不存在 → 解析错误

### 3.4 校验规则汇总

| 校验项 | 错误码 | 检测方法 |
|--------|--------|---------|
| ID 重复 | `TREELOCK_ERR_INVAL` | hash set 遍历 |
| parent 指向不存在的节点 | `TREELOCK_ERR_INVAL` | hash set 查找 |
| 无根节点 (parent==0 不存在) | `TREELOCK_ERR_INVAL` | 计数检查 |
| 多根节点 (parent==0 多于 1) | `TREELOCK_ERR_INVAL` | 计数检查 |
| 存在环路 | `TREELOCK_ERR_INVAL` | DFS + 访问标记 |
| 节点 label 重复 | 警告（不阻止） | hash set 查找 |
| JSON 语法错误 | `TREELOCK_ERR_INVAL` | cJSON 返回 NULL |

/* treelock_tree_internal.h — 树结构内部头文件 */

### 4.3 Hash 表设计（简单但够用）

节点数通常在几百到几万量级。使用**固定桶数的链式 hash**：
- `HASH_BUCKETS = 256`（可调）
- `hash_func(node_id) = node_id % HASH_BUCKETS`
- 冲突用 `next` 指针链起来
- 与现有 `lock_table` 的链表模式保持一致，后续可统一升级为 uthash

```c
#define TREE_HASH_BUCKETS   (256)

static treelock_tree_node_t *tree_hash_find(
    treelock_tree_node_t **table,
    treelock_node_id_t      node_id)
{
    UINT_32 bucket = (UINT_32)(node_id % TREE_HASH_BUCKETS);
    treelock_tree_node_t *node = table[bucket];
    while (node) {
        if (node->node_id == node_id) return node;
        node = node->next;
    }
    return NULL;
}
```

## 5. 核心算法

### 5.1 嵌套 JSON → 扁平节点列表

```
算法：flatten_tree(cjson_node, parent_id, output_list)
─────────────────────────────────────────────────────
输入: cJSON 树节点, 父节点 ID, 输出节点列表
输出: 成功/失败

1. 读取 node.id, node.label
2. 如果 id 已在 seen_ids 中 → 报错 "duplicate id"
3. 创建 treelock_tree_node_t, 设置 parent_id
4. 加入 output_list, 加入 seen_ids
5. 遍历 node.children:
6.     递归 flatten_tree(child, node.id, output_list)
7. 返回成功
```

时间复杂度：O(n)，n = 节点数。每个节点恰好访问一次。

### 5.2 树校验 — 环路检测

```
算法：detect_cycle(nodes, count)
─────────────────────────────────────
输入: 节点列表, 节点数
输出: TRUE（无环）/ FALSE（有环）

1. 构建 parent_map: node_id → parent_id
2. 对每个节点执行 DFS 路径追踪：
3.     visited = {}    ← 当前路径上的节点
4.     current = node
5.     while current.parent_id != 0:
6.         if current.id in visited → 发现环，返回 FALSE
7.         visited.add(current.id)
8.         current = parent_map[current.parent_id]
9.         if current == NULL → parent 不存在（单独报错）
10. 返回 TRUE
```

时间复杂度：O(n × d)，d = 树深度。对于典型的深度 < 100 的树，近似 O(n)。

### 5.3 协议自动校验 — `treelock_validate_protocol()`

```
算法：treelock_validate_protocol(tl, node_id, requested_mode)
─────────────────────────────────────────────────────────────
功能: 在加锁前自动检查用户是否已持有祖先节点的必要意向锁

1. if tree_index.loaded == FALSE → 跳过校验，返回 OK（向后兼容）
2. 获取 parent_id = tree_lookup(tl, node_id).parent_id
3. if parent_id == 0 → 根节点无父，返回 OK
4. 计算需要的最小父锁: required = treelock_required_parent_mode(requested_mode)
5. 检查: held_mode = treelock_get_mode(tl, parent_id)
6. if held_mode >= required → 返回 OK
7. else → 返回 TREELOCK_ERR_PROTOCOL
```

**关键设计决策**：如果树未加载（`loaded == FALSE`），协议校验直接放行。这保证了向后兼容——没有 JSON 文件的用户代码不受影响。

### 5.4 路径加锁 — `treelock_lock_path()`

```
算法：treelock_lock_path(tl, path, mode)
──────────────────────────────────────────
示例: path="/home/alice", mode=X
期望: lock(/,IS) → lock(home,IX) → lock(alice,X)

1. 解析 path: 按 '/' 分割 → ["", "home", "alice"]
2. 从根遍历树，收集路径上的所有 node_id:
3.     current = root
4.     path_ids = [root.id]
5.     for each segment:
6.         child = find_child_by_label(current, segment)
7.         if child == NULL → 返回 TREELOCK_ERR_INVAL
8.         path_ids.append(child.id)
9.         current = child
10. 对路径上每个节点加锁:
11.     for i = 0 to path_ids.len - 2:          ← 中间节点（祖先）
12.         ancestor_mode = IS  (如果 mode∈{IS,S})
13.                     或 IX  (如果 mode∈{IX,SIX,X})
14.         treelock_lock(tl, path_ids[i], ancestor_mode)
15.     treelock_lock(tl, path_ids[last], mode) ← 目标节点
16. 返回 TREELOCK_OK
```

### 5.5 路径解锁 — `treelock_unlock_path()`

```
算法：treelock_unlock_path(tl, path)
──────────────────────────────────────
1. 同 lock_path 解析路径，收集所有 node_id
2. 自底向上释放（与加锁顺序相反）:
3.     for i = path_ids.len - 1 down to 0:
4.         treelock_unlock(tl, path_ids[i])
5. 返回 TREELOCK_OK
```

## 7. 与现有代码的集成点

### 7.1 在 `treelock_lock()` 中嵌入协议校验

当前 `_do_lock_core()` 的流程：

```
参数校验 → 已持有检查 → 获取锁表节点 → 冲突检查 → 授予/等待 → 记录持有
```

需要在**冲突检查之前**插入协议校验：

```c
// _do_lock_core() 第 3 步之后，第 4 步之前插入：
/* ── 协议校验（树结构管理）── */
rc = treelock_validate_protocol(tl, node_id, mode);
if (rc != TREELOCK_OK) {
    return rc;  // TREELOCK_ERR_PROTOCOL
}
```

**改动量**：`client.c` 中 `_do_lock_core()` 增加 5 行。

### 7.2 在 `treelock_unlock()` 中嵌入释放顺序校验

当前 `treelock_unlock()` 的流程：

```
查找持有锁 → 从锁表移除 → 唤醒等待者 → 从持有列表删除
```

需要在**释放前**检查是否有未释放的子节点：

```c
// treelock_unlock() 开头，参数校验之后插入：
/* ── 释放顺序校验（树结构管理）── */
if (tl->tree_index.loaded) {
    rc = treelock_validate_release_order(tl, node_id);
    if (rc != TREELOCK_OK) {
        return rc;  // TREELOCK_ERR_PROTOCOL
    }
}
```

**改动量**：`client.c` 中 `treelock_unlock()` 增加 7 行。

## 9. 可行性评估

### 9.1 技术可行性 — ⭐⭐⭐⭐⭐

| 维度 | 评价 | 依据 |
|------|------|------|
| **代码改动范围** | 极小 | 现有模块仅 `client.c` 增加~12行，`protocol.c` 增加~30行，`internal.h` 增加 1 个字段 |
| **新增代码量** | ~800 行 | tree_core (~200) + tree_json (~200) + tree_validate (~100) + tree_path (~150) + 头文件 (~50) + cJSON (~1000 行但第三方) |
| **向下兼容** | 完全兼容 | 树未加载时协议校验跳过；所有现有 API 签名不变 |
| **第三方依赖** | 极轻量 | cJSON 仅 2 个文件，MIT 协议，无外部依赖，已在无数 C 项目中验证 |
| **性能影响** | 可忽略 | 协议校验 = 1 次 hash 查找 + 1 次锁模式比较；树加载 = O(n) 一次性操作 |

### 9.2 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| cJSON 解析大文件性能 | 低 | 目标场景节点数 10²~10⁵，cJSON 完全胜任 |
| 嵌套 JSON 递归栈溢出 | 中 | 限制最大深度（如 256）；或用显式栈替代递归 |
| label 路径查找 O(n) | 低 | label → id hash 表，O(1) 查找 |
| 现有链表锁表 O(n) 影响协议校验 | 低 | 校验操作本身只查 tree_index（hash），不遍历 lock_table |

### 9.3 结论

**方案可行**。核心改动仅 ~800 行新增代码 + ~50 行现有代码修改，不改动任何公共 API 签名，完全向后兼容。cJSON 集成零风险（2 文件、MIT 协议、数万项目使用）。实施周期预计 **2-3 天**（16 小时）。

## 附录 A：用户使用示例

### A.1 最简示例

```c
#include "treelock.h"
#include "treelock_tree.h"

int main()
{
    treelock_t *tl = treelock_create(NULL);

    // 一行加载树
    treelock_load_tree_from_file(tl, "my_tree.json");

    // 一行加锁：自动 root(IS) → home(IX) → alice(X)
    treelock_lock_path(tl, "/home/alice", TREELOCK_X);

    // ... 安全地修改 alice 的数据 ...

    // 一行解锁
    treelock_unlock_path(tl, "/home/alice");

    treelock_destroy(tl);
    return 0;
}
```

### A.2 编程式构建（不需要 JSON 文件）

```c
treelock_t *tl = treelock_create(NULL);

// 手动注册树结构
treelock_register_node(tl, 1, 0,  "/");         // 根
treelock_register_node(tl, 2, 1,  "home");       // /home
treelock_register_node(tl, 10, 2, "alice");      // /home/alice

// 协议自动生效！忘记锁父节点会直接报错
RET_CODE rc = treelock_lock(tl, 10, TREELOCK_X);
// rc == TREELOCK_ERR_PROTOCOL ← 没有先锁 / 和 /home
```
