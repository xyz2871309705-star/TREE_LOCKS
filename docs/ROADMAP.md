## 阶段一：单机版库 （目标：功能完备、可直接集成）

### Iteration 1.1 ✅ 已完成 — 核心协议

> 产出: 锁模式、兼容矩阵、模式转换

- [x] 6 种锁模式定义 (NL/IS/IX/S/SIX/X)
- [x] 兼容矩阵 (6×6 查表，O(1))
- [x] 锁升级路径验证 (15 条合法路径)
- [x] 锁降级路径验证
- [x] 工具函数 (mode_name / strerror)

### Iteration 1.2 ✅ 已完成 — 锁表与等待队列

> 产出: 内存锁表、FIFO 等待队列

- [x] 节点锁状态管理 (链表/可升级 uthash)
- [x] 授权列表 (grant_list) 与租约过期检查
- [x] FIFO 等待队列 + 条件变量唤醒
- [x] 冲突检查 (兼容矩阵 + 同客户端跳过 + 过期跳过)

### Iteration 1.3 ✅ 已完成 — 客户端 API

> 产出: 线程安全的 lock/unlock/try_lock

- [x] 生命周期管理 (create/destroy)
- [x] 阻塞锁 (treelock_lock)
- [x] 超时锁 (treelock_try_lock)
- [x] 锁升级/降级 (escalate/downgrade) — 原地更新模式，避免双重 entry
- [x] 同客户端重入加锁 (引用计数 ref_count)
- [x] 查询接口 (get_mode / query_node)

### Iteration 1.4 ✅ 已完成 — 基础工程

> 产出: 日志、跨平台、测试、文档

- [x] 统一日志模块 (6 级 + 外部回调)
- [x] 跨平台构建 (CMake + 平台抽象层)
- [x] 协议正确性测试 (12 个用例)
- [x] 并发压力测试 (3 个场景)
- [x] 开发者文档 (DEVELOPER.md)

### Iteration 1.5 🔲 待开始 — 树结构管理

> 目标: 库内部维护父子节点关系，执行协议规则 1/2/3

| 任务 | 预估工时 | 说明 |
|------|---------|------|
| 1.5.1 树节点注册 API | 3h | `treelock_tree_add(parent, child)` / `treelock_tree_remove(node)` |
| 1.5.2 父节点查询 | 1h | 内部 hash 表: node_id → parent_id |
| 1.5.3 协议自动校验 | 4h | lock 时自动查父节点，不满足 IS/IX 规则则拒绝 |
| 1.5.4 释放顺序校验 | 2h | unlock 时检查是否有子节点仍持锁（自底向上规则） |
| 1.5.5 树结构测试 | 3h | 深层嵌套、多叉树、动态增删节点 |

**验收**: `treelock_lock(tl, child, X)` 但未持有 parent 的 IX → 返回 `TREELOCK_ERR_PROTOCOL`

### Iteration 1.6 🔲 待开始 — 租约与故障恢复

> 目标: 即使单机版，也模拟租约机制，为分布式做准备

| 任务 | 预估工时 | 说明 |
|------|---------|------|
| 1.6.1 租约定时器 | 3h | 后台线程定期扫描过期锁 |
| 1.6.2 过期自动释放 | 2h | 租约到期 → 自动 unlock → 唤醒等待队列 |
| 1.6.3 心跳续约 API | 2h | `treelock_heartbeat(tl, node_ids[], count)` |
| 1.6.4 锁丢失回调触发 | 2h | 租约过期 → 调用 treelock_lost_cb |
| 1.6.5 租约测试 | 2h | 模拟心跳超时、回调验证 |

**验收**: 设置 500ms 租约 → 不续约 → 500ms 后锁被自动回收 → 回调被触发

### Iteration 1.7 🔲 待开始 — 性能与健壮性

> 目标: 基准测试、边界条件、内存安全

| 任务 | 预估工时 | 说明 |
|------|---------|------|
| 1.7.1 基准测试 | 3h | lock/unlock 吞吐量 (ops/s)、延迟分布 (p50/p99) |
| 1.7.2 哈希表升级 | 3h | 链表 → uthash (header-only)，节点查找 O(1) |
| 1.7.3 压力测试 | 2h | 10 万节点、1000 并发线程、长时间运行 (1h+) |
| 1.7.4 内存泄漏检测 | 2h | valgrind / AddressSanitizer |
| 1.7.5 边界测试 | 3h | NULL 参数、重复 destroy、超大型 node_id、并发 escalate |

**验收**: valgrind 0 leak，10 万节点下 lock 延迟 < 10μs

### Iteration 1.8 🔲 待开始 — API 完善

> 目标: 补齐缺失 API，提升易用性

| 任务 | 预估工时 | 说明 |
|------|---------|------|
| 1.8.1 批量锁 API | 3h | `treelock_lock_batch(tl, nodes[], modes[], count)` |
| 1.8.2 条件锁 | 2h | `treelock_lock_if_free(tl, node, mode)` — 非阻塞 |
| 1.8.3 锁信息导出 | 2h | `treelock_dump(tl, FILE*)` — 调试用完整状态导出 |
| 1.8.4 超时全局默认 | 1h | `treelock_set_default_timeout(tl, ms)` |
| 1.8.5 C++ 包装验证 | 2h | 确认 `extern "C"` 在 C++ 项目中正常工作 |

**验收**: 所有新增 API 有对应测试用例

> 目标: 基于 ZK 实现跨进程锁协调

## 迭代优先级排序

按优先级从高到低排列接下来的工作：

| 优先级 | 迭代 | 说明 |
|--------|------|------|
| 🔴 P0 | **1.5** 树结构管理 | 这是锁协议完整性的核心，不加锁协议规则形同虚设 |
| 🔴 P0 | **1.7** 哈希表升级 | 当前 O(n) 链表查找是性能瓶颈 |
| 🟡 P1 | **1.6** 租约机制 | 为分布式做准备，且单机版也实用 |
| 🟡 P1 | **1.7** 基准测试 + 内存检测 | 确保质量和性能基线 |
| 🟢 P2 | **1.8** API 完善 | 增强易用性 |
| 🟢 P2 | **2.1** ZK 集成 | 进入分布式阶段 |
| ⚪ P3 | 阶段三各迭代 | 自研服务版，工作量大 |
