# fix_recode — 代码风险修复记录

本目录存放所有代码审查后修复的风险点记录。每次 `/review-zh` 或 `/code-review` 后执行的修复，均须在此目录下创建对应的记录文件。

## 记录文件命名规范

```
FIX_RECORD_YYYYMMDD[_序号].md
```

示例:
- `FIX_RECORD_20250614.md` — 单次修复
- `FIX_RECORD_20250614_2.md` — 同日第二次修复
- `FIX_RECORD_20250615.md` — 次日修复

## 记录模板

每份记录必须包含以下结构：

```markdown
# 代码风险修复记录

**日期**: YYYY-MM-DD
**分支**: <branch_name>
**修复范围**: N 个风险项（X 高 + Y 中 + Z 低），涉及 M 个源文件

---

## 修复 N (🔴/🟠/🟡 风险等级): <简短标题>

**影响文件**: `path/to/file.c:line_range`
**问题**: 一句话描述风险点

### 修复方案

简要说明修复思路。

### 代码对比

```diff
-    // 修复前
+    // 修复后
```

---

## 测试结果

```
test_xxx:      N/N PASSED
test_yyy:      N/N PASSED
合计:          N/N PASSED (零警告编译)
```

## 变更文件清单

| 文件 | 变更类型 |
|------|----------|
| `path/to/file.c` | 描述变更内容 |
```

## 风险等级标签

| 标签 | 含义 |
|------|------|
| 🔴 高风险 | UB / 内存安全 / 数据丢失 / 死锁 / 协议违反 |
| 🟠 中等风险 | 竞态条件 / 错误处理缺失 / 资源泄漏 / 逻辑漏洞 |
| 🟡 低风险 | 注释不足 / 风格问题 / 可维护性改进 |
